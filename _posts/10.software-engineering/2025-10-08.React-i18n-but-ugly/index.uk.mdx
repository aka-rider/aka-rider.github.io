Або. Backend розробник розкаже вам за frontend

То жарти. Мета — показати, як розробити наколінне рішення вже тисячу разів вирішенної проблеми, яке має сенс.

## Маємо шо маємо

Переважно статичний сайт, React, Next.js App Router, markdown блог з декількома мовами.

**в наявності**: react-intl або i18next.

#### Проблема №1

`output: "export"` — означає відсутнє middleware, cookies, авто-визначення мови.

#### Проблема №2 (фатальна)

Те, що я "мушу зробити" — це створити та підтримувати:

```txt
messages-en.json
messages-es.json
messages-fr.json
....
```

#### Це означає

- Мені потрібно відтворювати структуру вебсайту в кожному з цих файлів
- Щоразу, коли треба додати, змінити або перемістити кнопку, мені потрібно одночасно змінювати всі ці `messages-${lang}.json`

#### Чисто практично

- Мені не те щоб кортіло показувати "X користувач-(ів) перебува-(є|ють) на сторінці"
- Я не віддаю переклади на аутсорс; мені не потрібні окремі фвйли
- Я можу використовувати LLM для створення принаймні початкових перекладів

На цьому кроці я хочу швиденько створити (може трохи кривенький) велосипед.
Мінімалістичне рішення щоб порівняти вартість підтримки.

#### Ключові умови

- **заморозити вимоги** — рішення максимально просте
- **прототип** — я не намагаюся спроектувати все наперед, я викидаю код і крокую вперед

## Винаходимо велосипед!

### Крок 1: зашити JSON у код

![](all-the-things.webp)

1. `messages-${lang}.json` є частиною додатку, а не конфігурацією (відсутнє значення ключа — це дефект).
2. Змушуємо компілятор TypeScript шукати втрачені переклади.
3. Описувати статичну структуру сайта, використовуючі систему типів TypeScript.

### Крок 2: запхнути все в один файл

```text
/i18n
└── index.ts
```

```typescript

const i18n = {
  en: {
    name: 'English',
    emoji: '🇬🇧',
    title: 'The website in English',
    .....
  },
  uk: {

```

### Крок 3: переписати існуючий фреймворк

```typescript
// Тип Lang виводиться з ключів структури i18n
export type Lang = keyof typeof i18n; // 'en' | 'uk'

// всі вкладені ключі всередині i18n:
// "en.title", "uk.title", "en.rootPage.heroCTA", ... і так далі
type Path<T> = T extends object
  ? {
      [K in keyof T]: K extends string ? K | `${K}.${Path<T[K]>}` : never;
    }[keyof T]
  : never;

type I18nPath = Path<(typeof i18n)[Lang]>;

export const getTranslations =
  (lang: string) =>
  (path: I18nPath): string => {
    if (!(lang in i18n)) return String(path);

    const keys = path.split('.');
    let i18node: any = i18n[lang as Lang];

    for (const key of keys) {
      if (i18node && typeof i18node === 'object' && key in i18node) {
        i18node = i18node[key];
      } else {
        return String(path);
      }
    }

    return typeof i18node === 'object' ? String(path) : String(i18node);
  };
```

З точки зору кінцевого користувача все виглядає як було.

Але тепер `getTranslations` з неіснуючим ключем може викликати помилку компіляції. Крім того, ми можемо вимагати, щоб ключ був визначений у всіх мовах.

Може те, може це — то вибір між поступовим перекладом сайту та суворою перевіркою на кожному кроці.

```typescript
// приклад використання
export default function Footer({ lang }: { lang: Lang }) {
  const t = getTranslations(lang);
  const currentYear = new Date().getFullYear();

  return (
    <footer>
      <p> © {currentYear} {t('title')} </p>
    </footer >
  );
}
```

### сумнівне рішення вже вирішеної проблеми: "існує" — що далі?

Я показую проміжні результати. Важливий крок — зупинитися та зробити переоцінку.

В цьому плані, моє рішення не те щоб...

#### ПЕРЕМОГИ:

- невідомі ключі викликають помилки компіляції
- не треба постійно стрибати поміж декількома JSON файлами

#### ПІДВОДНІ КАМЕНІ:

- жирний файл з гілковатими структурами — це не маштабується
- я перевинайшов колесо, і те криве, тепер мушу підтримувати

## Остаточне рішення

### Я не те що вирішую i18n — скоріше відокремлення контенту статичного сайту

Я намагаюся відокремити контент статичного вебсайту від HTML та CSS.
Тобто те, чим я вважав `i18n`, є чимось типу "Зе Древо."
Прилаштоване до будь якої форму контенту: інтернаціональний текст, світлини чи ікони.

### Не треба мімікрірувать фреймворк

Видалити `getTranslations(lang)`,

замість цього прямо: `i18n['lang'].title` — так компилятору простіше звітувати.

Без `getTranslations` жорстка структура: `<lang>: { <subtree> }` вже непотрібна.

### Натомість

```typescript
import { FaHome } from 'react-icons/fa';

export const navData = [
  {
    title: {
      en: "Home",
      uk: "На головну",
    },
    icon: FaHome,
    href: "/",
  },

  ...

] as const;

```

Далі все розмістити по різних файлах, повторюючи структуру сайту.
Що реданується разом, має жити разом.

#### Було

```text
/i18n
└── index.ts (Єдиний!)
```

#### Стало

```text
/i18n
└── common-components.ts
└── nav-panel.ts
└──/root-page
   └──index.ts
   └──hero-section.ts
...
```

#### Аут оф скоуп

Готові фреймворки краще підходять для динамічно генерованого контенту.
Я не кажу що мій підхід найкращий, але ж я можу згенерувати `messages-${lang}.json` якщо буде треба.

<TLDR title="Всі так роблять — то ще не привід" >

- фреймворки можуть бути тягарем для статичного сайту
- суворі рамки тримають рішення мінімально можливим — YAGNI
- ітерація замість ідеального дизайну
- [демонструю цей самий сайт](https://github.com/aka-rider/aka-rider.github.io)

</TLDR>
