У світі динамічних мов про Makefiles ходять легенди. І здебільшого — несправедливі.
Багато JS та Python розробників дивляться на GNU Make як на музейний експонат епохи палеозою, яким користуються хіба що бородаті адміни та C++ розробники.

А дарма. Make — це геніальний у своїй простоті інструмент.
І, до речі, половина того Linux чи macOS тримається саме на ньому.

Make заслуговує на ренесанс. І зараз я поясню нашо він потрібен твоєму проєкту на Python.

## Навіщо Python здався цей Makefile?

Коротка відповідь: щоб позбутися ситуації «Але ж у мене локально все працює!».

Так, Python чудово абстрагує нас від операційної системи. Але будь-який додаток — це більше, ніж просто код.
Це віртуальні оточення, залежності, конфігурація, докер-контейнери, тести, лінтери...

Зазвичай це закінчується купою bash-скриптів. І тут починається пекло.
Замість того, щоб пам'ятати, чи запускати `./setup_env.sh`, а потім `./run_app.sh` (чи може спочатку `./install_deps.sh`?), і згадувати, чи там підкреслення, чи дефіс у назві, можна ж просто:

```bash
$ make run
```

## Трохи про make

Якщо ви знайомі з make, сміливо пропускайте цей розділ.

Make — це інструмент, який відстежує залежності між файлами та їхніми мітками часу.
Якщо один з файлів-залежностей змінюється, make знає, що потрібно перезібрати файли, які від нього залежать.

Це надзвичайно корисно, коли мова йде про компіляцію коду, але не тільки.
Можна наприклад генерувати .jpg з .dot файлів, або .mpeg4 з окремих відео, аудіо та субтитрів, або .html файлів з .j2 чи .pug шаблонів.

Навіть сам Python робить щось подібне з .pyc файлами.

Правила в Makefile мають такий вигляд:

```make
target: dependency1, dependency2, …
<TAB>recipe line
<TAB>recipe line
…
```

target — це файл, який має зʼявитися;
dependency1 та dependency2 — це файли-залежності, що використовуються для створення target, а recipe — це просто команда командної строки, що використовується як крок збірки.

Припустимо, задача — просто скопіювати файл. Тоді правило виглядатиме так:

```make
dist/config.yaml: src/config.yaml
    cp -f src/config.yaml dist/config.yaml
```

Шановий, яке це все має відношення до Python?

Давайте розглянемо простий приклад.

Уявімо додаток, який показує випадкові гіфки з котами (*приклад взятий з* [Elm tutorial](https://elm-lang.org/examples/cat-gifs))

```python
# Додаток для відображення випадкових гіфок з котами
from http.server import HTTPServer, BaseHTTPRequestHandler
from io import BytesIO
import requests
class Handler(BaseHTTPRequestHandler):
    template = '''
<!DOCTYPE HTML>
<html>
<head><meta charset="UTF-8"><title>Cats</title></head>
<body>{body}</body>
</html>'''
    def do_GET(self):
        url = 'https://api.giphy.com/v1/gifs/random?api_key=dc6zaTOxFJmzC&tag=cat'
        r = requests.get(url)
        self.send_response(r.status_code)
        self.end_headers()
        if r.status_code == 200:
            data = r.json()
            body = '<img src="{imgurl}"/>'.format(imgurl=data['data']['image_url'])
            content = Handler.template.format(body=body)
        else:
            body = '<h1>HTTP - {code}</h1>'.format(code=r.status_code)
            content = Handler.template.format(body=body)
        self.wfile.write(content.encode('utf-8'))
port = 8000
httpd = HTTPServer(('localhost', port), Handler)
print("Відкрий http://localhost:{}/, щоб побачити щось цікаве".format(port))
httpd.serve_forever()
```

Тепер треба показати цей додаток друзям. Ні, **весь світ** повинен побачити його у всій красі.

Але виникає проблема:

```bash
Traceback (most recent call last):
 File “./app.py”, line 3, in <module>
 import requests
 ModuleNotFoundError: No module named ‘requests’
```

Зрозуміло, що простий `python3 app.py` не спрацює.

Давайте почнемо з віртуального середовища (venv) і встановимо в ньому модуль requests.

Для цього створимо файл `requirements.txt` зі спискоим залежностей:

```txt
requests
```

(Довгий, доо-овгий список.)

(Цей приклад я вигадував на ходу, тому він може бути не зовсім реальним).

Тепер давай створимо віртуальне середовище:

```bash
python3 -m venv venv
./venv/bin/pip install -r requirements.txt
```

Ми можемо навіть зберегти це в якомусь файлі `build.sh`.

Тепер все, що нам потрібно, це:

```bash
./venv/bin/python3 app.py
```

І ми також можемо захотіти зберегти цей однорядковий скрипт у якомусь `run.sh`.

Отже, це дуже поширена ситуація, про яку я говорив на початку: нам потрібно запустити build.sh перед run.sh, і ця залежність прихована і ніде не зазначена.

Давайте подивимося, чи стане ця штука з make зрозумілішою.

Першим правилом у нашому Makefile зробимо побудувати віртуальне середовище.

Ми могли би зробити щось на кшталт:

```make
venv: requirements.txt
    python3 -m venv venv
    ./venv/bin/pip install -r requirements.txt
```

... але `venv` — це каталог (папка), а для Makefile потрібен звичайний файл (час останньої модифікації каталога не має сенсу), і таке правило буде завжди виконуватися.

тому давайте візьмемо будь-який файл з «venv». Наприклад, `venv/bin/activate` підійде:

```make
venv/bin/activate: requirements.txt
    python3 -m venv venv
    ./venv/bin/pip install -r requirements.txt
```

Тепер, якщо ми запустимо make у каталозі проекту, ми отримаємо збірку віртуального середовища зі встановленим requests.
У нас це було раніше, з bash, але тепер залежність відстежується, тому запуск make вдруге дасть нам:

```bash
$ make
make: `venv/bin/activate’ is up to date.
```

Якщо ми відредагуємо `requirements.txt` — додамо або видалимо залежності, make автомагічно перезбере віртуальне середовище для нас:

```bash
$ touch requirements.txt  # simulate edit
$ make
python3 -m venv venv
./venv/bin/pip install -r requirements.txt
Requirement already satisfied: requests in ./venv/lib/python3.7/site-packages…
```

Тепер давайте створимо другу ціль під назвою run для запуску нашого додатку:

```make
run: venv/bin/activate
    ./venv/bin/python3 app.py
```

`run` явно залежить від цілі віртуального середовища, тому тепер вам потрібно лише ввести make run, і залежності будуть створені або оновлені перед запуском додатку.

Поки що все добре, але давайте розглянемо це детальніше.

По-перше, у нас немає файлу з назвою run; make думає, що цей файл буде створено після виконання рецепту, тому все працює.
Хоча, якщо хтось вирішить створити файл з назвою run, поведінка make стане несподіваною.

Давайте виправимо це, явно вказавши, що ми не очікуємо створення файлу.

```make
.PHONY: run
```

Тепер run — це фіктивна ціль, і відповідне правило буде виконуваться на кожен виклик (саме те, що нам потрібно).

Ми можемо захотіти додати ціль clean, щоб видалити те, що нам не потрібно:

```make
clean:
    rm -rf venv
    find . -type f -name ‘*.pyc’ -delete
```

Не забудьте, що ціль `clean` також не створює файл:

```
.PHONY: run clean
```

За замовчуванням, запуск make без аргументів виконає перше правило з Makefile.
Вважається хорошим тоном визначати вхідну ціль під назвою `all`, яка повинна визначати поведінку за замовчуванням.
У нашому випадку вона створить віртуальне середовище.

```make
all: venv/bin/activate
```

І знову:

```
.PHONY: all run clean
```

## Хочеш ще більше make?

Тут деякі люди проводять межу, але make також має змінні, функції та макроси.

Змінні дійсно прості:

```make
NAME := VALUE
```

визначає змінну, на яку можна посилатися як `${NAME}` або `$(NAME)`

Також є кілька корисних скорочень:

```make
NAME ?= VALUE
```

Присвоїти `VALUE` змінній `NAME`, якщо вона **ще не визначена**.

Це особливо корисно, тому що ви можете визначати змінні під час виконання make:

```bash
$ make BUILD=RELEASE ENVIRONMENT=PRODUCTION
```

Також є можливість оголосити псевдонім, ось так:

```make
NAME = VALUE
```

У цьому випадку права частина буде обчислюватися кожного разу при доступі до змінної.

Це можна легко пояснити на наступному прикладі

```make
v1 := $(shell ls)
v2 = $(shell ls)
```

Обидві змінні містять результат виконання команди оболонки ls, але v1 буде обчислено лише один раз і ніколи не покаже нових файлів.

До речі, shell — це функція make. Ура, ми щойно вивчили функції make!

Давайте підсумуємо все, що ми вивчили:

```make
# визначаємо назву каталогу віртуального середовища
VENV := venv

# ціль за замовчуванням, коли make виконується без аргументів
all: venv

$(VENV)/bin/activate: requirements.txt
	python3 -m venv $(VENV)
	./$(VENV)/bin/pip install -r requirements.txt

# venv — це скорочена ціль
venv: $(VENV)/bin/activate

run: venv
	./$(VENV)/bin/python3 app.py

clean:
	rm -rf $(VENV)
	find . -type f -name '*.pyc' -delete

.PHONY: all venv run clean
```

Легко, правда?

## Що ще?

Ви можете захотіти додати свій улюблений лінтер і форматер: `pep8`, `yapf`, `pyflakes` — що завгодно.
Ви можете згенерувати gRPC або swagger заглушку, ви можете згенерувати документацію з `.rst` або `.md` файлів.

Ось ще одна цікава можливість: паралельне виконання.

Можна запустити `make -j N` (j — кількість одночасних jobs) для паралельного виконання незалежних цілей.

Уявімо що ми хочемо конвертувати всі зображення в каталозі `assets/img` у формат `.webp`.

```make
IMG_DIR := assets/img
QUALITY := 80
# -m 6: найкраща якість стиснення
CWEBP_FLAGS := -q $(QUALITY) -m 6

#  Шукаємо всі наявні зображення (png, jpg, jpeg)
PNG_SRCS  := $(wildcard $(IMG_DIR)/*.png)
JPG_SRCS  := $(wildcard $(IMG_DIR)/*.jpg)
JPEG_SRCS := $(wildcard $(IMG_DIR)/*.jpeg)

# Створюємо список вихідних файлів webp, замінюючи розширення
# Синтакс: $(Variable:OldExtension=NewExtension)
# Якщо у `assets/img` у нас три файла: one.png, two.jpg, three.jpeg ...
# Змінна $(WEBPS) міститиме: (assets/img/one.webp, assets/img/two.webp, assets/img/three.webp)
WEBPS := $(PNG_SRCS:.png=.webp) \
         $(JPG_SRCS:.jpg=.webp) \
         $(JPEG_SRCS:.jpeg=.webp)

# Правило за замовчуванням all залежить від усіх webp файлів, які ми очікуємо на виході
all: $(WEBPS)
	@echo "Build complete! Converted $(words $(WEBPS)) images."

# Нам не потрібно записувати кожен файл, який ми хочемо конвертувати. Замість цього використовуються шаблонні правила.
# Ці правила кажуть make, як конвертувати будь-який .jpg у .webp і так далі
# Оскільки у нас немає конкретних імен файлів у шаблонних правилах, ми використовуємо Автоматичні Змінні make
# $< — Ім'я файлу першої залежності.
# $@ — Ім'я файла-цілі правила.
#
# Подробніше https://www.gnu.org/software/make/manual/html_node/Automatic-Variables.html

%.webp: %.png
	cwebp $(CWEBP_FLAGS) "$<" -o "$@"
%.webp: %.jpg
	cwebp $(CWEBP_FLAGS) "$<" -o "$@"
%.webp: %.jpeg
	cwebp $(CWEBP_FLAGS) "$<" -o "$@"

clean:
	rm $(WEBPS)

.PHONY: all clean
```

Для прикладу вище, запуск `make -j 4` дозволить конвертувати до 4 зображень одночасно.

Зверніть увагу на використання `@echo` у правилі all. make виводить кожну команду перед її виконанням. `echo драсті` виведе

```shell
$ make
echo драсті
драсті
```

Символ `@` перед командою запобігає виводу самої команди — виводиться тільки результат її виконання.

### Новий блискучий інструмент

Чому б не використовувати make для всього? Буквально для всього?!

Це дуже хороший інструмент для відстеження залежностей між файлами, але коли цілі динамічні, як-от docker-образи або запущені процеси... Ну, все може стати потворним і непередбачуваним. Кажуть пороблено.

<TLDR title='Makefiles все ще рулять!' />

Не соромтеся використовувати make за межами світу C/C++, отримуючи задоволення, радість і насолоду.

Кожного разу, коли вам потрібно створити файл з інших файлів, GNU make є топовим кандидатом для цієї роботи.
