Дякую, що клікнули. Дайте мені 3 хвилини, і я обіцяю — нудно не буде.

ООП — це зло? Можливо, ви вже погодилися, відкриваючи цю статтю. А можливо, ви тут лише для того, щоб переконатися, що автор — ідіот.
Моя теза проста: фраза *"ООП — це зло"* — це корисна ментальна модель. Навіть якщо ви фанат патернів, математичний пурист або просто прийшли сюди писати код за гроші.

Ця тема муляє мені вже давно, і ось чому:

1.  Я хочу читати чистий, красивий код.
2.  Я хочу, щоб софт, яким я користуюся, був залізобетонно надійним.
3.  Я хочу загинути в славетному холіварі (tabs vs spaces, імперативне vs декларативне, мікросервіси vs моноліт), щоб потрапити у Вальгаллу програмістів, де мене зустрінуть Алан Тюрінг та Ада Лавлейс.

Сучасне ООП (і викладання теми) — це парадигма, збудована навколо класів. Це суттєва проблема.

## Клас — це абстракція

Абстракцій у фізичному світі не існує. Це наш спосіб спростити реальність, щоб не збожеволіти.

**Змінна** — це абстракція (так, змінних теж не існує, це просто адреса в пам'яті).
Ми витрачаємо години на вибір ідеального імені для змінної, замість того щоб написати:

```assembly
mov [0xFA42BE6709A0F340], dword 5
```

_...і жити далі. Але це тема для іншого холівору._

Інша абстракція, яку ми використовуємо підсвідомо, і, на мою думку, вона не отримує достатньо уваги, — це **функція**.

Отже, клас — це абстракція над змінними та функціями. Це абстракція над абстракцією. Inception.

Повернімося до функції на хвилину. Попри видавану простоту, функції далеко не прості.
Нас вчать цих розмитих правил: хороша функція не повинна мати більше 5 параметрів, хороша функція повинна поміщатися на одному екрані...
Немає чіткого визначення, що таке хороша функція. Я бачив, до чого можуть призвести ці правила — 25 рядків коду, нова функція, поїхали.

На цьому етапі, я сподіваюся, ви розумієте, до чого я веду. Якщо ми не можемо рішуче впоратися з «простою» функцією, як ми можемо одночасно жонглювати дюжиною, пов'язаних не лише вводом-виводом, але й станом, який вони спільно змінюють?

Ніби цього було недостатньо, чого нас вчать? Що має провести нас крізь туман до світла? — Правильно! Поліморфізм, Успадкування та Інкапсуляція.

Ви жартуєте? Мій мозок плавиться, і я повинен отримати більше класів, тепер з перевантаженнями та перевизначеннями методів, і різними смаками множинного успадкування?

_Чому `BasketItem` успадковується від `MouseMovementObserverFactory`? Боже, що таке `MouseMovementObserverFactory`? Мені обіцяли об'єкти реального світу. Мені показали отика, собачку і загальний клас `Тваринка`._

Не зрозумійте мене неправильно. Це все дуже корисні концепції. Вони були створені, щоб усунути складність. Вони перевірені часом. Проблема в тому, що ООП саме по собі додає ще один шар складності.

**Ми додаємо складність, щоб усунути складність**. Ви бачите, як ця гра може мати несподівані наслідки?

## Класи: бути чи не бути?

> Я винайшов термін «Об'єктно-Орієнтований», і можу сказати вам, що я не мав на увазі C++. — Алан Кей

Алан Кей надихався біологією. У його баченні об'єкти — це окремі клітини, що спілкуються через **повідомлення** (messages).
Головним була саме ця комунікація (Actor Model).
Натомість, ми зациклилися на класах, статичних типах та ієрархіях. Ми будуємо мертві таксономії замість живих екосистем.

Проектування програмного забезпечення з використанням класів — це завжди програшна гра. Введення абстракцій перед реальною проблемою додає витрат без переваг.

Емпіричне правило — ніколи не починати з класів. Пишіть код, використовуючи лише функції, і класи проявляться самі собою, як фотографія Polaroid. Якщо вони взагалі потрібні.

_Група функцій, що отримують однакові аргументи, є солідним кандидатом на клас._

При проектуванні програмного забезпечення ключовим питанням є **не те, які сутності** складають систему, а **як сутності взаємодіють**.

Взаємодія — це власне код. Сутності, будь то класи, об'єкти чи функції — це бойлерплейт, щоб уможливити код, щоб допомогти нам. Не дозволяйте абстракціям забруднювати код.

Почніть з моделювання взаємодій, можливо, напишіть кілька фрагментів коду, просто зверху вниз, і сутності розкриються.

Якщо ми говоримо про проектування програмного забезпечення в термінах UML, емпіричне правило таке — ніколи не використовуйте діаграми класів UML для проектування програмного забезпечення.

![Example UML class diagram](https://cdn-images-1.medium.com/max/800/0*LqZfwGKKw6XDCsF2.png)

Вони описують сутності, але вони не описують взаємодію, тому вони не ведуть до надійного дизайну.

Завжди використовуйте діаграми послідовності (взаємодії) UML для моделювання програмного забезпечення.

![Example UML sequence diagram](https://cdn-images-1.medium.com/max/800/0*0ruSLR0OgEiyM2YN.png)

## Успадкування та Поліморфізм

Як і багато інших крутих концепцій та алгоритмів з Computer Science, і успадкування, і поліморфізм мають дуже обмежене застосування в реальному світі.

Я знаю, я теж обурений — я думав, що буду сортувати двосторонні колоди за O(log n), використовуючи генетичні алгоритми.
Правда в тому що сьогодні ми читаємо і пишемо JSON. _засмучений_робот_із_Рік_і_Морті.jpg_.

Важливо пам'ятати, що **Успадкування (IS-A)** є окремим випадком **Композиції (HAS-A)**.

Успадкування:

```cpp
A: B {
}
```

Композиція:

```cpp
A {
    B b;
}
```

Це означає, що кожне успадкування є композицією, тоді як не кожна композиція є успадкуванням. Тому ми звужуємо наші потенційні варіанти підтримки коду, обираючи успадкування заздалегідь.

_Загадки на кшталт того, чому BasketItem успадковується від MouseMovementObserverFactory, з'являються саме через це._

Розглянемо класичний приклад зоомагазину: Собака успадковує Тварину. Але чому?

Чи справді вони мають спільні методи?

Частіше за все, те, що ми бачимо по всьому реальному коду:

```javascript
if (pet.type == Cat) {
  // do this
} else if (pet.type == Dog) {
  // do this other thing
}
```

Потім хтось додає Золоту Рибку, і все летить шкереберть — місяці важкого рефакторингу.

Чи справді нам потрібно шукати серед усіх тварин? Чи користувачам справді байдуже, хочуть вони кота чи собаку?

Порівняйте код:

```javascript
for (pet in pets) {
  if (pet.type == Cat) {
    // but does it purr?
    (pet as Cat).purr();
  }
}
```

З:

```javascript
for (cat in cats) {
  // purrrrryty code...
}
```

## TL;DR

- ООП — це дуже потужна парадигма, але оскільки вона постачається без інструкції з використання, частіше за все вона призводить до надмірно ускладненого коду.
- Функція > Клас.
- Просто пишіть код, хороші абстракції розкриють себе, як фотографії Polaroid, включаючи класи.
- Використовуйте UML sequence діаграми для проектування, використовуйте діаграми класів для документації існуючого коду.

Заголовок каже про ООП, хоча в основному я писав про класи в популярних мовах програмування.
